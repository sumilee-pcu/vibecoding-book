# 13강. 종합 프로젝트와 책임

아이디어에서 출시까지, 한 사이클을 끝까지 돌리고 그 흔적을 남깁니다.

## 학습 목표

- 아이디어에서 설계·구현·검증·하네스화까지 전체 사이클(full cycle)을 한 번에 돌릴 수 있다.
- 이 책에서 익힌 안티패턴(anti-pattern) 총정리를 자신의 프로젝트에 적용해 점검할 수 있다.
- 보안·권한·개인정보·프롬프트 인젝션(prompt injection)·라이선스(license)까지 책임 소재를 따져 출시 전 점검을 할 수 있다.

## 이 장 한눈에

- 마지막 강은 새 기능을 잔뜩 배우는 자리가 아닙니다. 지금까지 익힌 것을 한 사이클로 꿰는 자리입니다.
- 평가 기준은 기능 수가 아니라 모든 단계가 남았는가입니다. 명세·완료 기준·테스트·git·하네스가 흔적으로 남아야 합니다.
- 책임은 출시 직전이 아니라 처음부터 설계에 들어갑니다. 권한·비밀 키·개인정보·프롬프트 인젝션 방어를 미리 규칙으로 둡니다.
- 안티패턴은 한 군데서 다시 모읍니다. "이렇게 하면 망한다"를 출시 전에 한 번 더 비춰 봅니다.
- 다음 단계로 잇습니다. 여기서 닫은 사이클이 더 큰 프로젝트의 한 칸이 됩니다.

## 개념

### 13.1 종합 프로젝트: 아이디어에서 출시까지 한 사이클

종합 프로젝트는 큰 앱을 만드는 일이 아닙니다. 작은 기능 하나라도 아이디어에서 출시까지 한 바퀴를 끝까지 돌리는 일입니다. 사이클의 칸은 명세 쓰기, 규칙 정하기, 첫 기능의 완료 기준·테스트, Plan Mode 계획과 승인, 한 조각씩 구현, diff 검토, 통과 단위 커밋, 보안·권한·라이선스 점검입니다. 한 칸이라도 비면 그 자리에서 사이클이 끊깁니다.

### 13.2 도메인과 데이터 선정

먼저 무엇을 만들지 한 문장으로 좁히고, 다룰 데이터를 정합니다. 데이터는 공개·가짜 데이터로 시작합니다. 실제 개인정보나 비밀 키는 실습 저장소에 두지 않습니다. 이 장의 종합 프로젝트는 8~10강에서 만든 도서 메모 검색 도구를 토대로 삼습니다. 도메인이 익숙하면 사이클의 칸마다 무엇이 빠졌는지 더 잘 보입니다.

### 13.3 설계에서 하네스화까지

설계는 명세(SPEC)에서 시작합니다. 이번 범위와 제외 범위, 완료 기준, 자동 확인과 수동 확인, 남은 위험을 적습니다. 구현은 한 조각씩 합니다. 한 조각을 만들고 자동 확인을 돌려 통과하면 커밋합니다. 마지막으로 이 흐름을 하네스(harness)로 굳혀, 다음에도 같은 순서로 반복할 수 있게 합니다. 하네스화(harnessing)는 좋은 결과를 운에 맡기지 않고 틀로 받치는 일입니다.

### 13.4 안티패턴 총정리

이 책에서 본 안티패턴을 한자리에 모읍니다. 판단까지 위임하기, 한 방에 다 시키기, 맥락 없이 탓하기, 이해 없이 고치기, 느낌으로 끝내기, 검증 없는 배포·되돌릴 수 없는 변경, 형식주의 하네스, 무인 자동화·과잉 자동화입니다. 출시 전에 이 목록을 한 번 더 비춰 봅니다. 자세한 점검은 `checklists/안티패턴_총정리.md`에 있습니다.

### 13.5 책임 있는 바이브코딩: 보안·권한·개인정보·프롬프트 인젝션

생성은 도구가 하더라도 책임 소재는 사람에게 있습니다. 비밀 키와 토큰은 저장소에 두지 않고, 도구에는 최소 권한(least privilege)만 줍니다. 외부 문서·웹 검색 결과·MCP 응답 안에 섞여 들어온 지시는 실행하지 않습니다. 이것이 프롬프트 인젝션 방어입니다. 쓰기·삭제·배포·외부 전송은 사람 승인 뒤에만 합니다. 민감 데이터(sensitive data)는 출력하거나 외부로 보내지 않습니다.

### 13.6 라이선스와 책임 소재

새로 들여온 라이브러리·자산은 라이선스를 확인합니다. AI가 만든 코드라도 책임은 그것을 출시한 사람에게 있습니다. 출처가 분명치 않은 코드·데이터·이미지를 그대로 출시하지 않습니다. 책임 소재를 분명히 해 두면, 문제가 생겼을 때 누가 무엇을 점검해야 하는지가 명확합니다.

### 13.7 출시 전 점검과 다음 단계

마지막은 출시 전 점검(release check)입니다. 완료 기준이 모두 테스트로 묶였는지, 마지막 테스트 실행 결과가 남았는지, diff에 범위 밖 변경이 없는지, 라이선스·권한·비밀 키·프롬프트 인젝션 방어를 확인했는지 봅니다. 점검은 이 폴더의 `RELEASE_CHECK.md`를 그대로 따릅니다. 여기서 닫은 사이클은 다음 프로젝트, 그리고 LLM 애플리케이션으로 잇는 출발점이 됩니다.

## 실습

### [프로젝트] 실전 앱 1개 완성·배포

**목표:** 8~10강의 도서 메모 검색 도구에 기능 하나만 더해, 아이디어에서 출시까지 모든 단계가 흔적으로 남게 합니다.

**준비물:** AI 코딩 도구 1종, 8~10강에서 만든 도서 메모 검색 도구(`filterNotes`), 빈 작업 폴더, 이 폴더의 체크리스트.

**더할 기능(아래 중 하나만 고릅니다):**

- 태그로 메모 거르기
- 즐겨찾기 표시와 즐겨찾기만 보기
- 도서 데이터 MCP 조회로 책 정보 채우기
- 검색 기록 남기고 다시 쓰기

**단계:**

1. 명세 — `SPEC.md`에 이번 범위·제외 범위·완료 기준·자동 확인·수동 확인·남은 위험을 적습니다.
2. 규칙 — `CLAUDE.md`에 작업 규칙과 프롬프트 인젝션 방어 규칙을 둡니다.
3. 완료 기준·테스트 — 더할 기능 하나의 완료 기준을 테스트로 적고, 체크리스트와 연결합니다.
4. 계획·승인 — Plan Mode로 계획을 받아 검토하고 승인합니다.
5. 구현 — 한 조각씩 구현하고 자동 확인(`node --test`)을 돌립니다.
6. 검토 — diff에서 범위 침범·테스트 완화가 없는지 봅니다.
7. 커밋 — 통과한 단위만 Git으로 커밋합니다.
8. 점검 — `RELEASE_CHECK.md`로 보안·권한·라이선스·민감 데이터·프롬프트 인젝션을 점검하고, 남은 위험을 README에 적습니다.

**완료 기준:**

- 더한 기능이 정상·경계·실패 경우에서 동작한다.
- 완료 기준이 모두 테스트로 묶여 있고, 마지막 테스트 실행 결과가 남았다.
- 명세(SPEC)·규칙(CLAUDE.md)·테스트·Git 커밋·하네스 순서·보안 점검이 모두 흔적으로 남았다.
- 비밀 키·개인정보가 저장소에 없다.

기준은 기능 수가 아니라 모든 단계가 남았는가입니다.

## 실행과 자기 평가

- 실행: 도서 메모 검색 도구 폴더에서 `node --test`로 테스트를 돌립니다.
- 평가: 이 폴더의 `checklists/표13-1_완료점검표.md`로 6단계를 자기 점검합니다.
- 출시 전: `RELEASE_CHECK.md`의 항목을 모두 통과해야 출시로 봅니다.

## 체크리스트

- ☐ 명세(SPEC.md)에 이번 범위·제외 범위·완료 기준·남은 위험을 적었다.
- ☐ CLAUDE.md에 작업 규칙과 프롬프트 인젝션 방어 규칙을 두었다.
- ☐ 더할 기능 하나의 완료 기준을 테스트로 적었다.
- ☐ Plan Mode로 계획을 받아 승인한 뒤 구현했다.
- ☐ 한 조각씩 구현하고 자동 확인을 돌렸다.
- ☐ diff에서 범위 밖 변경이 없는지 확인했다.
- ☐ 통과한 단위만 Git으로 커밋했다.
- ☐ RELEASE_CHECK.md의 보안·권한·라이선스·개인정보·프롬프트 인젝션을 점검했다.
- ☐ 남은 위험을 README에 적었다.

## 핵심 용어

- 전체 사이클(full cycle) — 아이디어·설계·구현·검증·하네스화·출시까지 한 바퀴.
- 하네스화(harnessing) — 좋은 결과를 운에 맡기지 않고 반복 가능한 틀로 굳히는 일.
- 프롬프트 인젝션(prompt injection) — 외부 문서·검색 결과·MCP 응답 안에 숨은 지시로 도구를 조종하려는 공격.
- 최소 권한(least privilege) — 도구·계정에 일을 끝낼 만큼의 권한만 주는 원칙.
- 책임 소재(accountability) — 결과에 대한 책임이 누구에게 있는지. AI 코드라도 출시한 사람에게 있다.
- 민감 데이터(sensitive data) — 개인정보·비밀 키·내부 자료처럼 출력·외부 전송을 막아야 할 데이터.
- 출시 전 점검(release check) — 배포 전에 완료 기준·테스트·보안·라이선스를 확인하는 마지막 관문.

## 연습문제

### 개념 확인

1. 종합 프로젝트의 평가 기준이 "기능 수"가 아니라 무엇인지 적어 보세요.
2. 전체 사이클의 여덟 칸(명세부터 점검까지)을 순서대로 적어 보세요.
3. 프롬프트 인젝션이 무엇인지, 방어 규칙 한 가지를 적어 보세요.
4. 최소 권한 원칙을 한 문장으로 설명해 보세요.
5. 출시 전 점검 항목 중 보안·민감 데이터와 관련된 것 두 가지를 적어 보세요.

### 서술형

1. AI가 만든 코드라도 책임 소재가 사람에게 있다는 말의 뜻을, 라이선스 점검과 묶어 설명해 보세요.
2. "기능을 하나만 더한다"가 종합 프로젝트의 규칙인 이유를, 모든 단계가 남아야 한다는 기준과 연결해 설명해 보세요.

### 실습형

1. 도서 메모 검색 도구에 위 네 기능 중 하나만 더해, `SPEC.md`·`CLAUDE.md`·테스트·Git 커밋·`RELEASE_CHECK.md`가 모두 흔적으로 남도록 한 사이클을 완성해 보세요.
