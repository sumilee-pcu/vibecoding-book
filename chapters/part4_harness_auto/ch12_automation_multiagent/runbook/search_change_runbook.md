# 검색 기능 변경 런북

도서 메모 검색 기능을 바꿀 때 따르는 8단계입니다. 메인 에이전트가 흐름을 이끌고, 세 서브에이전트가 역할을 나눠 일합니다.

## 8단계

① 메인이 목표와 완료 기준을 확인한다. SPEC.md에서 이번 변경이 연결되는 완료 기준을 적는다.
② code-explorer가 관련 파일·흐름·위험 지점을 요약한다. 원본은 바꾸지 않는다.
③ 메인이 계획을 검토하고 승인한다. 수정 후보를 3개 이내로 확정한다.
④ safe-implementer가 승인된 파일만 수정한다. 범위를 넘으면 멈춘다.
⑤ test-runner가 테스트를 실행하고 실패만 요약한다.
⑥ 실패가 있으면 한 원인만 골라 다시 구현한다(④로 돌아간다).
⑦ 통과하면 메인이 diff를 검토한다. 범위 침범과 테스트 완화를 확인한다.
⑧ 사람 승인 뒤 커밋 후보를 제안한다.

## 단계별 입출력

| 단계 | 담당 | 입력 | 출력 |
| --- | --- | --- | --- |
| ① | 메인 | SPEC.md | 이번 변경의 완료 기준 |
| ② | code-explorer | 완료 기준 | 관련 파일·흐름·위험·수정 후보 |
| ③ | 메인(+사람) | 수정 후보 | 승인된 계획 |
| ④ | safe-implementer | 승인된 계획 | 변경 파일·이유·영향 범위 |
| ⑤ | test-runner | 변경 결과 | 통과/실패 요약 |
| ⑥ | safe-implementer | 실패 원인 1개 | 재수정 |
| ⑦ | 메인 | diff | 범위 침범·테스트 완화 검토 결과 |
| ⑧ | 메인(+사람) | 검토 결과 | 커밋 후보 |

## 런북 실행 프롬프트

도서 메모 검색 기능을 바꾸려고 해. 이 런북의 8단계를 따라 진행해 줘.
먼저 SPEC.md에서 이번 변경의 완료 기준을 확인하고, code-explorer로 관련 파일을 조사해 요약만 가져와 줘.
계획은 내 승인 뒤에만 구현하고, safe-implementer는 승인된 파일만 수정해 줘.
구현 뒤 test-runner로 테스트를 실행하고 실패만 요약해 줘.
실패가 있으면 한 원인만 골라 다시 구현해 줘. 통과하면 diff를 검토하고, 커밋은 내 승인 뒤 후보만 제안해 줘.
