# 하네스의 세 요소

하네스(harness)는 세 가지로 이루어집니다. 할 일, 완료 기준, 자동 확인입니다. 이 셋이 갖춰지면 모델은 정해진 틀 안에서 움직이고, 사람은 결과를 명세와 대조하기만 하면 됩니다.

## ① 할 일: 명세(SPEC.md)

무엇을 만들지 적습니다. 목표, 제약, 완료 기준을 포함합니다.

- 목표: 이번 작업이 이루려는 것.
- 제약: 바꾸면 안 되는 것, 지켜야 하는 것.
- 완료 기준: 무엇이 충족되면 끝인지.

명세가 흔들리면 결과도 흔들립니다. 그래서 코드보다 명세가 먼저 옵니다.

## ② 완료 기준: 관찰 가능한 조건

완료 기준은 "잘 동작한다" 같은 막연한 말이 아니라, 보면 통과 여부를 알 수 있는 조건이어야 합니다.

- (약함) 검색이 잘 된다.
- (강함) 제목 일부로 검색하면 해당 메모가 결과에 나온다.

관찰 가능한 조건은 그대로 테스트로 옮길 수 있습니다.

## ③ 자동 확인: 테스트·린트·타입 검사

자동 확인은 사람이 매번 눈으로 보지 않아도 통과 여부를 알려 줍니다.

- 테스트: 완료 기준이 실제로 충족되는지 확인.
- 린트(lint): 코드 스타일과 흔한 실수를 확인.
- 타입 검사(type check): 값의 종류가 어긋나지 않는지 확인.

자동 확인이 있으면 완료 기준이 깨졌을 때 즉시 드러납니다.

## 명세 기반 개발(SDD)

명세 기반 개발(Spec-Driven Development, SDD)은 코드를 쓰기 전에 명세를 먼저 확정하는 방식입니다. 순서는 다음과 같습니다.

명세 확정 → 완료 기준을 테스트로 고정 → 구현 → 자동 확인 → 명세와 결과 대조.

이 순서를 지키면 모델이 임의로 범위를 넓히기 어렵고, 사람은 매번 같은 기준으로 결과를 검토할 수 있습니다.
