# 10강. 나만의 작업 하네스 구축

프롬프트 한 줄에 기대지 않고, 같은 품질을 반복해 내는 작업 환경을 만듭니다.

## 학습 목표

- 하네스(harness)를 정의하고 하네스를 이루는 세 요소를 설명할 수 있습니다.
- 계획 모드(Plan Mode)·승인 절차·규칙 파일을 하네스의 구성 요소로 배치할 수 있습니다.
- 명세(SPEC)·규칙·기록을 잇는 반복 가능한 파이프라인(pipeline)을 설계할 수 있습니다.

## 이 장 한눈에

| 개념 | 다루는 내용 |
| --- | --- |
| 10.1 하네스란 무엇인가 | 프롬프트가 아니라 환경으로 품질을 고정한다 |
| 10.2 하네스의 세 요소 | 할 일(명세)·완료 기준·자동 확인 |
| 10.3 최소 하네스 | SPEC.md·CLAUDE.md·RUNLOG.md 세 파일 |
| 10.4 명세 기반 개발(SDD) | 코드보다 명세가 먼저 온다 |
| 10.5 파이프라인과 검사 지점 | 슬래시 명령으로 흐름을 굳힌다 |
| 10.6 하네스 안티패턴 | 과설계·형식주의·멈춤 없는 반복 |

10.1 하네스란 무엇인가. 하네스는 모델이 일관된 품질로 작업하도록 묶어 주는 환경입니다. 도구·규칙 파일·승인 절차·기록이 모여 하나의 작업 틀을 이룹니다. 좋은 프롬프트는 한 번의 결과를 좌우하지만, 좋은 하네스는 매번의 결과를 좌우합니다.

10.2 하네스의 세 요소. 하네스는 할 일(명세), 완료 기준, 자동 확인으로 이루어집니다. 할 일은 무엇을 만들지 적은 명세입니다. 완료 기준은 무엇이 충족되어야 끝인지 적은 관찰 가능한 조건입니다. 자동 확인은 사람이 매번 눈으로 보지 않아도 통과 여부를 알려 주는 테스트·린트·타입 검사입니다.

10.3 최소 하네스. 처음부터 큰 틀을 짤 필요는 없습니다. 세 파일이면 시작할 수 있습니다. SPEC.md에 목표와 완료 기준을 적고, CLAUDE.md에 규칙을 적고, RUNLOG.md에 매 작업의 기록을 남깁니다. 이 세 파일이 최소 하네스(minimal harness)입니다.

10.4 명세 기반 개발(SDD). 명세 기반 개발(Spec-Driven Development, SDD)은 코드를 쓰기 전에 명세를 먼저 확정하는 방식입니다. 명세가 흔들리면 결과도 흔들립니다. 명세가 분명하면 모델은 그 안에서 움직이고, 사람은 명세와 결과를 대조하기만 하면 됩니다.

10.5 파이프라인과 검사 지점. 파이프라인은 명세 읽기에서 시작해 탐색·계획·구현·테스트·검토·커밋으로 이어지는 정해진 흐름입니다. 검사 지점(checkpoint)은 그 흐름 중간에서 통과 여부를 확인하는 자리입니다. 슬래시 명령(slash command)으로 파이프라인을 묶어 두면 매번 같은 순서를 따를 수 있습니다.

10.6 하네스 안티패턴. 하네스는 과하면 오히려 일을 막습니다. 과설계, 형식주의, 모델만 믿기, 사람을 흐름 밖으로 밀어내기, 멈춤 없는 반복은 피해야 할 다섯 가지 패턴입니다. 자세한 내용은 `checklists/하네스_점검.md`에서 다룹니다.

## 실습

[실습] 최소 하네스 템플릿

목표: 도서 메모 검색 도구(filterNotes) 프로젝트에 최소 하네스를 적용합니다. 프로젝트 루트에 SPEC.md·CLAUDE.md·RUNLOG.md 세 파일을 두고, 슬래시 명령으로 안전한 변경 흐름을 묶습니다.

단계:

① 이 장의 `SPEC.md`를 프로젝트 루트로 복사하고, 도서 메모 검색 도구의 목적·제약·완료 기준을 자기 프로젝트에 맞게 채웁니다.
② `CLAUDE.md`를 루트에 두고, 테스트 명령과 금지 행동을 자기 환경에 맞게 확인합니다.
③ `RUNLOG.md`를 루트에 두고, 첫 작업을 한 건 기록합니다. 실패와 수정이 한 줄이라도 들어가야 합니다.
④ `.claude/commands/safe-change.md`를 두고 `/safe-change`를 한 번 실행해 파이프라인이 도는지 확인합니다.

완료 기준:

- SPEC.md에 목표·범위·완료 기준·자동 확인이 모두 적혀 있습니다.
- CLAUDE.md에 테스트 명령과 금지 행동이 적혀 있습니다.
- RUNLOG.md에 최소 1회의 작업 기록(실패와 수정 포함)이 있습니다.
- `/safe-change`가 승인 전에 코드를 바꾸지 않습니다.

## 체크리스트

- ☐ SPEC.md에 목표·범위·완료 기준이 모두 있다.
- ☐ CLAUDE.md에 테스트 명령과 금지 행동이 있다.
- ☐ 자동 확인을 실제로 한 번 실행해 보았다.
- ☐ 멈추는 기준(stop condition)을 정해 두었다.
- ☐ RUNLOG.md에 실패와 수정이 최소 1회 기록되었다.
- ☐ 커밋 전 diff로 범위 침범을 확인했다.
- ☐ 하네스가 과설계가 아니다(세 파일로 시작했다).

## 핵심 용어

- 하네스(harness): 모델이 일관된 품질로 작업하도록 묶어 주는 환경.
- 최소 하네스(minimal harness): SPEC.md·CLAUDE.md·RUNLOG.md 세 파일로 시작하는 가장 작은 하네스.
- 자동 확인(automated check): 사람이 매번 보지 않아도 통과 여부를 알려 주는 테스트·린트·타입 검사.
- 명세 기반 개발(Spec-Driven Development, SDD): 코드보다 명세를 먼저 확정하는 개발 방식.
- 파이프라인(pipeline): 명세 읽기에서 커밋까지 이어지는 정해진 작업 흐름.
- 검사 지점(checkpoint): 흐름 중간에서 통과 여부를 확인하는 자리.
- 실패 경로(failure path): 테스트 실패·범위 침범 등 정해진 조건에서 멈추거나 사람에게 넘기는 경로.

## 연습문제

1. 하네스의 세 요소를 적고, 각각이 도서 메모 검색 도구에서 무엇에 해당하는지 한 줄로 쓰세요.
2. 프롬프트 한 줄로 품질을 맞추는 방식과 하네스로 품질을 맞추는 방식의 차이를 두 문장으로 설명하세요.
3. 자기 프로젝트의 완료 기준 세 개를 관찰 가능한 조건으로 다시 쓰세요.
4. 하네스 안티패턴 다섯 가지 중 자기에게 가장 위험한 하나를 고르고 이유를 쓰세요.
