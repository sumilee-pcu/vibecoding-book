# 9강 프롬프트 모음 (verbatim)

버그를 잡고, 작게 고치고, Git으로 안전하게 되돌리는 흐름을 순서대로 담았습니다. 그대로 복사해 쓰세요.

## ① 버그 잡기 (재현 데이터 담은 버그 보고)

```
filterNotes(notes, query)에 버그가 있어. 버그 보고를 만들어 줄게.
증상: filterNotes(notes, " python ")가 빈 배열을 돌려준다.
기대: 앞뒤 공백을 무시하고 "python"과 같은 결과(2개)가 나와야 한다.
재현 데이터: data/sample-notes.json. 재현 방법: node --test.
이 증상의 원인을 코드에서 찾아 줘.
```

## ② 잘못된 코드 감별

```
src/filterNotes.buggy.js를 읽고, 완료 기준 6개와 어긋나는 부분을 모두 짚어 줘.
각 문제마다 "어떤 입력에서 어떻게 어긋나는지"를 구체적으로 말해 줘.
```

## ③ 작게 고치기 (한 원인만)

```
방금 찾은 문제 중 앞뒤 공백 버그 하나만 고쳐 줘.
다른 부분은 건드리지 말고, 정규화 한 곳만 최소한으로 바꿔 줘.
고친 뒤 node --test로 해당 테스트가 통과하는지 확인해 줘.
```

## ④ 변경 폭 제한

```
이번 수정은 src/filterNotes.buggy.js와 test/filterNotes.test.js 안에서만 해 줘.
data/나 README, 다른 파일은 바꾸지 마. 바꾼 파일 목록을 마지막에 알려 줘.
```

## ⑤ 안전 리팩터링 (normalizeText 추출)

```
정규화(trim + 소문자) 로직이 여러 군데 반복돼. normalizeText라는 도우미 함수로 추출해 줘.
동작은 그대로여야 해. 리팩터링 전후로 node --test 결과가 똑같이 전부 통과하는지 확인해 줘.
```

## ⑥ Git 안전망 (변경 전 상태 확인·diff·커밋 후보)

```
지금 작업 트리 상태를 git status로 확인해 줘.
바꾼 내용을 git diff로 보여 주고, 어떤 단위로 커밋하면 좋을지 커밋 메시지 후보를 제안해 줘.
```

## ⑦ diff 검토

```
방금 만든 변경의 git diff를 검토해 줘.
의도한 한 가지 수정만 들어 있는지, 관계없는 줄이 섞이지 않았는지 확인해 줘.
범위를 벗어난 변경이 있으면 어떤 줄인지 알려 줘.
```

## ⑧ 부분 복구 (storage.js만 되돌리기)

```
storage.js의 변경만 실수였어. 다른 파일은 그대로 두고 storage.js만 마지막 커밋 상태로 되돌려 줘.
git restore로 해당 파일만 복구하는 명령을 알려 주고, 되돌린 뒤 node --test로 다시 확인해 줘.
```
