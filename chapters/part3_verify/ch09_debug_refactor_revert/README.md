# 9강. 버그 고치고 Git으로 되돌려보기

## 학습 목표

- 증상·기대·재현을 담은 좋은 버그 보고를 쓸 수 있다.
- 잘못된 코드를 완료 기준과 대조해 감별할 수 있다.
- Git으로 커밋·diff·되돌리기를 하며 안전하게 고칠 수 있다.

## 시작하기: Git 초기화

이 폴더는 Git 실습용입니다. 처음 한 번 Git 저장소를 만들고 기준점을 커밋해 둡니다.

```
cd chapters/part3_verify/ch09_debug_refactor_revert
git init
git add .
git commit -m "test: add book note search acceptance tests"
```

이렇게 해 두면 무엇을 바꾸든 `git diff`로 보고 `git restore`로 되돌릴 수 있습니다.

## 핵심 개념

### 9.1 좋은 버그 보고

버그 보고는 세 가지를 담습니다. 증상(무슨 일이 일어났나), 기대(무엇을 기대했나), 재현(어떻게 다시 일으키나). 재현 데이터와 명령까지 적으면 AI도 사람도 같은 화면을 봅니다.

### 9.2 잘못된 코드 감별

버그는 완료 기준과의 차이로 드러납니다. "앞뒤 공백을 무시한다"는 기준이 있는데 `trim()`이 없으면, 그 한 줄이 버그입니다. 기준을 손에 들고 코드를 읽으면 어긋난 곳이 보입니다.

### 9.3 작게 고치기

한 번에 한 원인만 고칩니다. 버그 3종을 한꺼번에 손대면 무엇이 무엇을 고쳤는지 알 수 없습니다. 하나 고치고 테스트, 하나 고치고 테스트입니다.

### 9.4 Git 안전망

`git status`로 현재 상태를 보고, `git diff`로 바뀐 줄을 확인하고, 의미 단위로 `git commit` 합니다. 커밋은 언제든 돌아올 수 있는 저장 지점입니다.

### 9.5 diff 검토와 범위 침범

수정 후에는 diff를 읽어 "의도한 한 가지"만 바뀌었는지 확인합니다. 관계없는 파일이나 줄이 섞였다면 범위 침범입니다.

### 9.6 부분 복구

전체를 되돌릴 필요는 없습니다. `git restore <파일>`로 잘못 건드린 파일 하나만 마지막 커밋 상태로 되돌릴 수 있습니다.

## [실습] 버그 있는 앱 고치고 Git으로 되돌려보기

도서 메모 검색 도구의 `filterNotes`에는 버그 3종이 심겨 있습니다.

- 버그 ① 앞뒤 공백 미처리 (trim 누락)
- 버그 ② 대소문자 구분 (양쪽 정규화 누락)
- 버그 ③ 검색하며 원본 배열 직접 변경 (splice로 원본 훼손)

진행 방법:

1. `node --test`를 돌립니다. 버그 때문에 일부 테스트가 실패합니다.
2. 실패 하나를 골라 `debug-report-template.md`로 디버깅 리포트를 씁니다.
3. 버그 하나만 고치고 다시 `node --test`.
4. `git diff`로 범위를 확인하고 `commit-messages.md`의 예시처럼 커밋합니다.
5. ①~③을 한 번에 하나씩 반복합니다.

`prompts.md`의 ①~⑧ 프롬프트가 이 흐름을 그대로 따라갑니다.

## 실행법

```
cd chapters/part3_verify/ch09_debug_refactor_revert
node --test
```

처음 실행하면 버그 ①②③ 때문에 공백·대소문자·원본 불변 관련 테스트가 실패합니다. 이는 정상이며, 실패가 버그의 위치를 가리킵니다.

정답은 `src/filterNotes.fixed.js`를 참고하세요. 정답으로 전부 통과하는지 확인하려면 `test/filterNotes.test.js`의 import 경로를 `../src/filterNotes.fixed.js`로 바꿔 실행합니다.

## 완료 기준 (이 실습 자체의)

- 버그 ①②③을 각각 디버깅 리포트로 설명했다.
- 버그를 한 번에 하나씩 고치고 매번 커밋했다.
- 최종적으로 `node --test`가 전부 통과한다.

## 체크리스트

- ☐ 각 버그마다 증상·기대·재현을 적었는가
- ☐ 한 번에 한 원인만 고쳤는가
- ☐ 고칠 때마다 `node --test`로 확인했는가
- ☐ `git diff`로 범위 침범이 없는지 봤는가
- ☐ 의미 단위로 커밋했는가

## 핵심 용어

- 버그 보고: 증상·기대·재현을 담아 문제를 전달하는 글.
- 재현: 같은 입력으로 문제를 다시 일으키는 것.
- diff: 변경 전후의 차이.
- 커밋: 되돌아올 수 있는 저장 지점.
- 되돌리기: `git restore`/`git revert`로 이전 상태로 복구하는 것.
- 범위 침범: 의도하지 않은 파일·줄까지 바뀐 상태.

## 연습문제

1. 버그 ③(원본 변경)을 먼저 고치고 ①②를 나중에 고치면, 중간 단계에서 어떤 테스트가 통과/실패할지 예상해 보세요.
2. `git revert`와 `git restore`의 차이를 한 문장으로 설명해 보세요.
3. 세 버그를 한 커밋에 몰아 고쳤다고 가정하고, 그 커밋이 왜 위험한지 안티패턴 체크리스트와 연결해 설명해 보세요.
