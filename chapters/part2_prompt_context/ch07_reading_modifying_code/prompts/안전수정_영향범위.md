# 안전한 수정과 영향 범위

큰 변경일수록 한 번에 하지 않습니다. 영향 범위를 먼저 보고, 단계로 나눠 매번 확인하며 진행합니다.

## 대규모 변경 단계 분할
"이 프로젝트의 날짜 처리 방식을 전부 바꾸려고 한다. 한 번에 바꾸지 말고, 먼저 어떤 파일들이 영향을 받는지 목록과 변경 계획을 보여 줘. 내가 승인하면 한 파일씩 바꾸고, 매번 테스트를 돌려 통과를 확인한 뒤 다음으로 넘어가자."

이 요청의 안전장치:
- 영향 파일 목록과 계획을 먼저 본다.
- 사람이 승인한 뒤에 시작한다.
- 한 파일씩 바꾸고 매번 테스트로 확인한다.

## 영향 범위 확인
함수 하나를 바꾸기 전에, 그 함수를 누가 쓰는지 먼저 봅니다.
"getUser 함수를 바꾸려고 한다. 이 함수를 호출하는 곳이 어디어디인지 모두 찾아 줘. 각 호출처가 반환값을 어떻게 쓰는지도 알려 줘. 그래야 바꿔도 안전한지 판단할 수 있다."

노트: 호출처를 모르고 시그니처를 바꾸면, 멀리 떨어진 코드가 조용히 깨질 수 있습니다.

## 쓸모 있는 주석 요청
"이 함수에 주석을 달아 줘. 단, 코드만 봐도 아는 내용 말고, 왜 이렇게 처리했는지와 고칠 때 조심할 점 위주로 적어 줘."

노트: "i를 1 증가시킨다" 같은 주석은 군더더기입니다. 의도와 주의점을 남기는 주석이 나중에 도움이 됩니다.
