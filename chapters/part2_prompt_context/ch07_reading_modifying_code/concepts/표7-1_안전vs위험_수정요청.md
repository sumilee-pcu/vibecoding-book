# 표 7-1. 안전한 수정 요청 vs 위험한 수정 요청

같은 의도라도 요청 방식에 따라 안전성이 크게 달라집니다. 다섯 상황을 비교합니다.

| 상황 | [위험한 요청] | [안전한 요청] | 이유 |
| --- | --- | --- | --- |
| 큰 파일 수정 | "이 파일 전체 고쳐 줘." | "이 파일에서 X 함수만 바꾸고 나머지는 그대로 둬." | 범위를 좁혀야 예상 못 한 변경을 막는다 |
| 리팩터링 | "코드 구조 싹 바꿔 줘." | "중복된 A 로직 세 군데만 함수 하나로 합쳐 줘. 동작은 그대로." | 동작 보존을 명시해야 회귀를 막는다 |
| 버그 수정 | "버그 좀 고쳐 줘." | "이 에러 메시지가 뜨는 이 줄을, 입력이 없을 때도 안 멈추게 고쳐 줘." | 증상·위치를 줘야 엉뚱한 곳을 안 고친다 |
| 의존성 변경 | "라이브러리 최신으로 올려 줘." | "어떤 코드가 이 라이브러리를 쓰는지 먼저 찾고, 영향과 계획을 보여 줘." | 영향 범위를 먼저 봐야 깨짐을 예측한다 |
| 테스트 수정 | "실패하는 테스트 통과하게 해 줘." | "테스트가 왜 실패하는지 원인부터 알려 줘. 테스트가 맞으면 코드를 고친다." | 테스트를 무력화하지 않고 원인을 고친다 |

노트: 마지막 줄이 특히 중요합니다. 통과만 시키려고 테스트를 약하게 바꾸면, 버그가 그대로 남은 채 초록불만 켜집니다.
